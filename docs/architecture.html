Internal Operations Console — Architecture & Decisions

A lightweight, in-house alternative to a Retool-style platform: one console bundling the three internal apps the client actually uses (KYC review, refunds, feature flags), with real business logic, server-side RBAC, an immutable audit log, and mock vendor integrations.

PostgreSQL 16 FastAPI · SQLAlchemy 2 · Alembic React · Vite · TypeScript · Tailwind TanStack Query Monorepo

Two flows. Command path: the SPA calls /api/* with a cookie session → a thin router authenticates and checks permissions → the service layer runs the business rule inside a DB transaction, writes an audit event, and (for KYC/refunds) calls a provider adapter to write back to the vendor. Sync path (inbound): vendor data lands in the read-only integration_* source tables, and the sync/ETL layer normalizes those raw rows into the domain tables — an idempotent upsert that never overwrites operator-owned state. All rules live on the server; the UI only mirrors permissions for UX.

Data flow — vendor → staging → normalized domain

Vendor data is never used raw. It lands in a staging table (integration_*), and a sync/ETL step (app/services/sync_service.py) normalizes those rows into the domain tables the app actually runs on. The transform is idempotent (upsert by natural key) so it can be re-run for backfills and reconciliation, and it never overwrites operator-owned state.

What gets normalized & why: vendor status strings → canonical KycStatus/PaymentStatus enums (one vocabulary for the state machine + RBAC); raw 0-100 risk score → risk_level band (stable triage); money → integer minor units + single currency; the vendor inquiry is resolved to our customer by email; identifiers (vendor, vendor_reference_id, provider_payment_id) are kept for provenance, and the original JSON stays in raw_vendor_payload/raw for audit and re-normalization. Vendor states only ever reach NEEDS_REVIEW — the APPROVED/REJECTED decision is made by a reviewer inside the console. Ingestion is sync-only (no inbound webhooks); write-backs (issuing a refund, requesting more info) go outbound to the vendor via an adapter and are reconciled by the next sync.

Key decisions & tradeoffs

Python backend + React frontend (not the plan's Next.js/Prisma)

The client mandated Python + React. Chose FastAPI + SQLAlchemy 2 + Alembic for typed models, explicit migrations, and a clean place to put business logic. Tradeoff: two deployables and no end-to-end type sharing like a TS-only stack — accepted for language fit and a strong server-side domain layer.

Thin routers, fat service layer

Every rule (KYC state machine, refund limits/idempotency, flag versioning) lives in app/services/, not routers or React. Why: the UI can never be the source of truth for a fintech's money/identity operations; logic stays testable and reusable. Tradeoff: a little more indirection than inlining logic in endpoints.

Money as integer minor units

All amounts are cents (amount_minor), never floats. Why: avoids rounding errors in refunds and balance math. Tradeoff: formatting happens at the edges (UI + serializers).

Concurrency: row locks for refunds, optimistic versions for flags

Refunds take a SELECT … FOR UPDATE on the payment so concurrent refunds can't over-refund; status is recomputed from the DB after each success. Flags use an expected_version check returning 409 VERSION_CONFLICT. Why: refunds need a hard guarantee at write time; flag edits are low-contention and benefit from a friendlier optimistic model.

Feature flags: JSONB config per environment

Each (flag, env) stores {enabled, rollout_percentage, filters[]} in JSONB, with legacy booleans upgraded transparently. Why: matches the PostHog-style ask (rollout % + targeting) without a schema change per new config knob. Tradeoff: validation moves into the service (normalize_config) instead of the DB.

Idempotency & deterministic mock failures

Refunds require an idempotency key (a duplicate returns the original result); the mock payment provider fails deterministically when a provider id ends in FAIL. Why: lets the whole failure path be tested end-to-end with no real vendor.

Sync/ETL ingestion (staging tables), not webhooks

Vendor data lands in integration_* staging tables and a sync step (sync_service) normalizes it into the domain tables; idempotent upsert by natural key. Why over webhooks: the full table can be re-run for backfill/reconciliation, raw payloads are durable and inspectable, dedupe is trivial, and there's no inbound endpoint to secure — webhooks drop/duplicate events and only cover "after you subscribed." Tradeoff: latency (batch vs. real-time); acceptable for KYC/refund review, and production would run it on a schedule (often with webhooks as a low-latency trigger).

Immutable audit log for every mutation

KYC decisions, refunds, and flag changes all write append-only audit_events with before/after snapshots. Why: compliance and traceability are core for a fintech; it's cheap to add up front and painful to retrofit.

Dev-only auth (signed cookie + user switcher)

Seeded users with a role switcher, not real SSO. Why: keeps the prototype runnable and lets reviewers exercise RBAC instantly. Tradeoff / production gap: SSO/SAML + real provisioning is the first thing to add before cutover.

Database structure

PostgreSQL 16, managed with Alembic migrations. Every table has a string id primary key; most carry created_at/ updated_at timestamps. Money is stored as integer minor units (cents). The integration tables are the read-only staging source (raw vendor mirrors); the domain tables hold the normalized state the app runs on, produced by the sync/ETL layer.

Domain tables

TablePurposeNotable columns & rules
users Operators of the console. Unique email; role drives all RBAC. Referenced as KYC reviewer/decider, refund requester, flag editor, and audit actor.
customers End customers KYC & payments belong to. Unique email, country_code, date_of_birth. Parent of kyc_cases and payments.
kyc_cases The KYC review queue. status state machine + risk_level/risk_score; nullable assigned_reviewer_id/decided_by_id; decision_reason (required on reject); full vendor response in raw_vendor_payload (JSONB).
kyc_case_events Per-case transition history. from_status→to_status, event_type, actor, and JSONB event_metadata — append-only.
payments Charges eligible for refunds. amount_minor (cents) + currency; status (SUCCEEDED/PARTIALLY_/FULLY_REFUNDED/DISPUTED/FAILED); provider + provider_payment_id; masked payment_method_last4.
refunds Refunds issued against a payment. amount_minor, reason, status; unique idempotency_key (dedupes retries); failure_reason for provider failures. Sum of SUCCEEDED refunds drives the parent payment's status.
feature_flags Flag definitions. Immutable unique key; type, owner, tags[]; archived_at freezes edits.
feature_flag_values Current config per environment. Unique (flag_id, environment); JSONB value = {enabled, rollout_percentage, filters[]}; version powers optimistic concurrency.
feature_flag_versions Immutable change log per flag/env. previous_value→new_value (JSONB), version, required reason, changed_by_id.
audit_events Cross-cutting immutable audit trail. action, entity_type/entity_id, JSONB before/after, ip_address. Written for every KYC/refund/flag mutation.

Integration tables (staging source — raw vendor mirrors)

Read-only staging tables holding raw vendor records (seeded here; a batch pull in production). The sync/ETL layer reads them and normalizes into the domain tables. Each keeps a raw jsonb payload so the transform can evolve without schema churn; not foreign-keyed to the domain tables (they model an external source of truth).

TableStands in forNotable columns
integration_persona_inquiries Persona identity-verification inquiries (KYC source). Unique inquiry_id, reference_id, status, name/email, country_code, risk_score, raw.
integration_stripe_charges Stripe charges (payments/refunds source). Unique charge_id, payment_intent, amount/amount_refunded, currency, status, card brand/last4, raw.
integration_launchdarkly_flags LaunchDarkly flags (feature-flag source). Unique flag_key, name, kind, temporary, maintainer, per-env environments (JSONB), raw.
